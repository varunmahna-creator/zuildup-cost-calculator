### C. THE refresh bug — true root cause (round 3, ~11:36 UTC)
Varun sent two screenshots: searched "varun", URL=`?q=varun`, header said "2 total leads", but list rendered all 208 rows. Even after manual refresh, header updated but list didn't.

**Real root cause:** `LeadsListClient.tsx` line 94:
```js
const [leads, setLeads] = useState<Lead[]>(initialLeads)
```

`useState(initialValue)` only reads the initial value on first mount. When the Server Component re-renders with new filtered `initialLeads`, the Client Component receives the new prop but useState ignores it. Page header `totalCount` (direct prop, not state-derived) updates correctly. Local `leads` state stays frozen on the first-mount dataset.

This was THE bug behind **three** reported symptoms:
- **Bug 1** (filters not working) — URL changed, list state ignored prop update
- **Bug 2** (search no results visible) — header showed correct count, list state frozen
- **Bug 3** (inline-activity refresh required manual reload) — I patched `router.refresh()` three times yesterday + today. Refresh always fired. Client just ignored the new props because of the useState snapshot.

**Why this masqueraded as 3 bugs:** all my E2E tests were curl + DB query, which proved the SERVER side was correct every time. The bug was 100% client-side, only visible in real React mount cycle.

**Fix (commit `1fe0c21`, Vercel `dpl_Fe4fbuA6QoXoReN9SQoEgBBKNnof`):**
```js
const [leads, setLeads] = useState<Lead[]>(initialLeads)
useEffect(() => {
  setLeads(initialLeads)
}, [initialLeads])
```

One effect. Three bugs dead. Varun confirmed "resolved" at 11:40 UTC.

## 🚨 Lessons learned today (added to MEMORY.md as 🚨 LESSON 2026-05-29)

### Lesson 1: Never `useState(someProp)` in a client component without a sync effect
- `useState(initialValue)` ONLY reads `initialValue` on first mount.
- If a Server Component re-renders with a new prop, the Client Component receives the new prop but useState ignores it.
- **Symptom signature:** parent-level direct-prop UI (counts, headers) updates correctly; state-derived UI (lists, forms) appears frozen. Manual reload "fixes" it because remount re-seeds.
- **Fix:** either render directly from props (preferred), OR add `useEffect(() => setX(prop), [prop])` sync.
- **Anti-pattern is invisible in dev** because hot-reload remounts components. Only manifests after real server re-render in prod.
- **Code review rule:** any `useState(someProp)` in a Client Component is suspect. Justify it or remove it.

### Lesson 2: When adding role-scoped UI to an existing page, check middleware FIRST
- Vaishali's dashboard fix: I added KPI cards + verified API auto-scopes for her, but never checked whether middleware was letting her reach the page at all. Page-level code is necessary but NOT sufficient if middleware bounces the request before the page ever runs.
- **Rule:** before assuming a page-level bug, `grep -r "/<route>" middleware.ts auth.ts` for any role gate.

### Lesson 3: Multiple symptoms from one bug = stop and look one level up
- Round 1 (inline-activity refresh): patched startTransition → microtask. Failed.
- Round 2: patched microtask → await sequencing. Failed user-side. My E2E test (curl + DB) passed because it bypassed the client.
- Round 3 would have been another `router.refresh()` patch. STOPPED, re-read components, found the props-snapshot useState bug.
- **The 3-strike rule (MEMORY.md) saved a 4th wrong commit.**
- **Pattern recognition:** if 3 user-reported "different" symptoms (filters, search, refresh) all "fix" with manual reload, they probably share ONE client-side root cause. Look for state derived from a prop without sync.

### Lesson 4: "Curl + DB E2E" is necessary but NOT sufficient for frontend bugs
- All my E2E tests for bug 3 over two days were curl POST + DB SELECT. They always passed because the server was correct.
- The bug was a React mount-cycle issue. Only visible in a real browser with React reconciliation.
- **Rule:** when a bug is reported as "UI doesn't update", curl-and-DB proves the API works — but does NOT prove the page renders. For UI-render bugs, need either: live browser session, OR React component code-audit looking for state-from-prop antipatterns, OR a Playwright/Cypress test that asserts DOM state.
